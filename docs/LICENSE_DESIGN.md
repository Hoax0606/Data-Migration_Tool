# ModernizeProData 라이선스 설계 — 결정 사항 정리

> **이 문서의 목적**: ModernizeProData 라이선스 시스템 설계에 대해 합의된 결정 사항을 다른 Claude 환경에서도 그대로 이어갈 수 있도록 정리한 self-contained 메모. 본 문서만 읽어도 현 결정 상태를 이해할 수 있도록 작성.
>
> **상태**: 설계 합의 단계. V1 본 개발에서 구현 예정.
> **마지막 갱신**: 2026-05-11.

## 0. 한눈에 보는 요약

| 영역 | 결정 |
|---|---|
| **제품** | ModernizeProData (KS Info System 의 한국 SI 데이터 이행 도구) |
| **시장** | 한국 1·2 금융권·공공·SI 환경. 폐쇄망 가정. |
| **라이선스 단위** | **사이트 단위** (계약서로 인스턴스 한도 통제) |
| **검증 알고리즘** | **Ed25519 서명** + 공개키 도구 임베드, 오프라인 100% |
| **사이트 ID 부여** | **본사가 정함** (영업 주도, 사람이 읽기 쉬운 슬러그) |

| **인터넷 호출** | 0건 (라이선스 갱신은 .lic 파일을 본사가 전달) |

## 1. 배경 — 왜 이 설계인가

### 1.1 본 도구의 환경 가정

- **폐쇄망(에어갭) 가정** — 사이트가 인터넷에 접근 불가한 경우 많음. 라이선스 검증이 외부 서버 호출에 의존하면 동작 불가.
- **한국 SI 운영팀 친화** — 메인프레임 운영팀은 보수적. 도구가 사이트 인프라를 강제하거나 *PC 마다 fingerprint 활성화* 같은 마찰을 일으키면 도입 거부.
- **사이트 단위 영업** — KS Info System 영업은 *프로젝트 단위 / 사이트 단위* 계약이 표준. 사용자당·머신당 라이선스는 한국 SI 시장 비표준.
- **재설치·HA·DR 자유** — 운영팀이 도구를 PC 교체·HA 구성·DR 환경 추가할 때 *본사 재발급 없이* 같은 라이선스로 동작해야 함.

### 1.2 한국 SI 라이선스 시장 표준 (참고)

한국 SI 시장에서 통상적인 라이선스 관행:
- 영업 계약서가 1차 통제 (인스턴스 수·기간 명시)
- 도구는 *만료일 + 기능 flag* 만 강제, 머신 fingerprint 는 거의 안 씀
- 위반 시 *audit log 기반 영업·법무 대응*
- 외산 도구처럼 *PC 활성화 코드* 흐름은 한국 SI 에 거부감

→ 본 도구는 이 시장 관행을 따른다.

## 2. 검증 알고리즘 — Ed25519 서명

### 2.1 핵심 흐름

```
[본사 = 라이선스 발급자]
  ┌─ 비밀키 (private key) ── 본사 격리 보관 (HSM 권장)
  │
  └─ payload (JSON) 에 비밀키로 서명 → .lic 파일

[도구 = 라이선스 검증자]
  ┌─ 공개키 (public key) ── 도구 빌드 시 임베드
  │
  └─ .lic 받으면 공개키로 서명 검증
     · 서명 OK → payload 신뢰
     · 서명 깨짐 → 위변조 인식 → 거부
```

### 2.2 왜 Ed25519?

- 가볍고 빠름 (RSA 대비)
- 표준 라이브러리 (libsodium / Bouncy Castle / Web Crypto) 광범위 지원
- 서명 길이 짧음 (64 바이트)
- 키 생성·서명·검증 모두 단순

### 2.3 키 관리

| 키 | 보관 | 노출 | 분실 시 |
|---|---|---|---|
| **비밀키** (Private) | 본사 격리 보관 — HSM · 하드웨어 토큰 · 별도 서버 | **절대 외부 노출 금지** | 모든 라이선스 무효화 + 새 키 발급 + 모든 고객사 재발급 |
| **공개키** (Public) | 도구 코드에 임베드 | 노출 OK (공개해도 위조 불가) | 공개키 자체는 분실 개념 없음. 도구 빌드에 포함됨 |

→ 비밀키 유출이 *최악 시나리오*. HSM·OS 키링·격리된 USB 등 가능한 강한 보호.

## 3. .lic 파일 구조

### 3.1 형식

```
.lic 파일 = base64( payload_json + signature )
```

내부적으로:

```json
{
  "v": 1,
  "licenseId": "MPD-2026-0421-KDB",
  "customer": "KDB Bank",
  "siteId": "kdb-corebanking-2026",
  "edition": "enterprise",
  "features": ["multi-site", "external-trigger", "advanced-rules"],
  "issuedAt": "2026-04-21",
  "expiresAt": "2027-04-21",
  "graceDays": 14
}
```

별도 필드 `signature`: 위 payload 의 JCS(JSON Canonicalization Scheme) 정규화 + SHA-256 + Ed25519 서명값 (base64).

### 3.2 payload 필드 의미

| 필드 | 의미 |
|---|---|
| `v` | 라이선스 스키마 버전 (forward compat) |
| `licenseId` | 본사 발급 식별자 — 본사 내부 추적용 |
| `customer` | 고객사 표시 이름 (사람용 라벨) |
| `siteId` | **이 라이선스의 본 식별자** — 도구가 자기 ID 와 비교 |
| `edition` | `standard` / `pro` / `enterprise` |
| `features[]` | 활성화될 기능 flag 배열 |
| `issuedAt` | 발급일 |
| `expiresAt` | 만료일 (이 날까지 ACTIVE) |
| `graceDays` | 만료 후 grace 기간 (기본 14) |

### 3.3 *박지 않는 것* (의도적 배제)

- `machineFingerprint` — 머신 바인딩 안 함 (운영 친화)
- `maxInstances` — 인스턴스 한도는 *도구가 강제하지 않음*. 계약서·audit log 가 통제.
- `nonProdAllowed` — 사이트 내부 dev/staging 자유 (기본 허용)

→ 단순화. 사이트 ID + 만료일 + 기능 flag 만으로 통제. *나머지는 영업·계약 영역*.

## 4. 사이트 ID — 핵심 결정

### 4.1 부여 주체 — **본사가 정함** (영업 주도)

대안 비교:

| 방식 | 누가 ID 정함 | 채택? |
|---|---|---|
| **A. 본사 정함** | 영업이 계약 단계에서 부여 | **✓ 채택** |
| B. 도구 자동 생성 (UUID) | 도구 첫 실행 시 자체 생성, admin 이 본사에 보고 | ✗ (Microsoft 식. 한국 SI 친화도 ↓) |

A 채택 이유:
1. 한국 SI 영업 흐름 (계약 → 발급) 과 직선 매칭
2. 사람이 읽고 검증 가능 (UUID 보다 직관)
3. 도구 재설치·이전 시 *같은 ID 그대로* 사용 → 본사 재발급 X
4. 본사 통제력 ↑
5. 컴플라이언스 감사 시 *누가 누구의 라이선스인지* 한눈에

### 4.2 ID 형식 — 본사 표준

```
<고객사 슬러그>-<도메인 슬러그>-<발급 연도>[-<연차 분기>]
```

예시:
- `kdb-corebanking-2026`
- `nh-deposit-2026`
- `kt-billing-2025-q3`
- `hyundai-erp-2026`

규칙:
- 소문자만
- 하이픈 구분
- 공백·한글·특수문자 금지 (파일명·URL·로그 안전)
- 짧지만 *고객사 + 도메인 + 시기* 식별 가능

### 4.3 본사 ↔ 현장 연결 흐름

```
[본사]                           [고객사]

영업
  ↓ 계약 체결
  ↓ 사이트 ID 부여 (예: kdb-corebanking-2026)
  ↓
라이선스 발급팀
  ↓ .lic 생성 (Ed25519 서명)
  ↓ 설치 안내서 (사이트 ID 포함) 동봉
  ↓
  └→ sFTP / 사내 메일 / USB / 공문  ─→  admin
                                         ↓
                                        도구 설치
                                         ↓
                                        Solution Settings
                                         ↓ 사이트 ID 입력 (영업이 받아쓰기 검증)
                                         ↓ .lic 업로드
                                         ↓
                                        도구가 검증:
                                         · 서명 OK?
                                         · siteId 일치?
                                         · 만료일 안?
                                         · 시계 조작 없음?
                                         ↓
                                        활성 / 거부
```

**연결의 핵심 주체**: 본사 *영업 SE*. SE 가
- 사이트 ID 결정 + 발급팀에 요청
- .lic 받아 고객사에 전달
- 첫 설치 시 동석 (가능하면) — 사이트 ID 받아쓰기 검증
- 갱신 시 같은 ID 로 새 .lic 발급

## 5. 만료 정책 — 4 단계 라이프사이클

```
유효기간 종료 30일 전  ─ 인앱 알림 "30일 후 만료"
                         ↓
유효기간 종료일      ─ 화면 상단 빨간 배너 + 알림 강화
                         ↓
ACTIVE                   ↓ (expiresAt 까지)
                         ↓
IN_GRACE (14일)          ↓ 모든 기능 정상. 화면 상단 빨간 배너 유지.
                         ↓
READ-ONLY (15~30일)      ↓ run / cutover / 매핑 편집 차단. 조회·내보내기만.
                         ↓
완전 차단 (+31일~)       ↓ 모든 기능 차단. audit log 만 조회 가능.
```

### 5.1 갱신 흐름

만료 30일 전부터 *본사 영업 SE 가 갱신 영업* 시작:
1. 새 계약 체결 (또는 자동 연장)
2. 본사가 같은 사이트 ID 로 새 .lic 발급 (expiresAt 갱신)
3. SE 가 고객사 admin 에게 새 .lic 전달
4. admin 이 Solution Settings 에서 새 .lic 업로드 → 즉시 ACTIVE 갱신

### 5.2 read-only 단계의 의미

EXPIRED 후 15~30일은 *조회·내보내기만 허용*:
- 매핑 그리드 열람 OK
- Audit log 조회 OK
- 산출물(YAML / SQL / report) 다운로드 OK
- **신규 매핑 작성·실행·스냅샷 생성 X**

→ 운영 영향 최소화하면서 갱신 지연에 대한 *압박은 유지*.

## 6. 위협 모델 — 무엇을 차단하나

| 위협 | 차단 메커니즘 |
|---|---|
| **위조** (admin 이 만료일 손으로 늘리기) | 서명 검증 실패 → 거부 |
| **복사** (다른 회사가 .lic 사용) | siteId mismatch → 거부 |
| **시계 되감기** (PC 시간 변경) | 봉인된 last-seen 시각보다 이전 → CLOCK_TAMPERED 차단 |
| **만료 후 계속 사용** | grace 14일 후 read-only, 30일 후 완전 차단 |
| **공개키 교체로 위조 도장 통과** | 도구 무결성 검증 (코드 서명 / hash) 으로 공개키 변조 감지 |
| **인터넷 우회 시도** | 검증이 오프라인이라 인터넷 호출 자체가 없음. 우회 대상이 없음. |

### 6.1 차단하지 *않는* 것

- **같은 사이트 ID 로 다른 PC 에 라이선스 복사** — 도구는 fingerprint 안 박으므로 *기술적으로 차단 X*. 이는 *의도된 정책*. 같은 사이트 내 HA·DR·dev 자유.
- **계약 한도 초과 인스턴스 운영** — 도구는 막지 않음. *Audit log 에 instance fingerprint 가 자동 기록* 되므로 본사 영업이 갱신 협상 시 활용.
- **사이트 ID 사칭** (다른 회사가 'kdb-corebanking-2026' 입력) — 기술적으로는 활성되지만 *법적으로 KDB 은행으로 사칭한 행위*. audit log 가 *다른 회사 IP* 에서 활성된 흔적을 남기므로 본사 감사 시 적발 가능.

→ **기술적 차단** vs **계약·감사 통제** 의 균형. 한국 SI 운영팀 친화도를 우선.

## 7. 시계 조작 방지

### 7.1 메커니즘

도구가 마지막 검증 시각을 *암호화 봉인된 상태 파일* 에 기록:
```
<install>/data/license-state.bin   (AES-GCM 봉인)
```

검증 시 비교:
- 현재 시각 ≥ 봉인된 last-seen 시각 → 정상 진행
- 현재 시각 < 봉인된 last-seen 시각 → **CLOCK_TAMPERED 차단**

### 7.2 한계

- OS 전체 시각이 *항상 앞으로* 만 가는 환경에선 안전
- 단일 PC 에서 *시간만 잠깐 앞으로 → 다시 뒤로* 같은 조작은 last-seen 봉인 깨짐으로 잡힘
- HA 환경에서 NTP 가 정상 동기화되면 문제 없음
- 실제로 운영 환경에서 NTP 가 동기화되는데 *고의 조작* 자체가 흔치 않음

## 8. 도구 측 동작 (구현 노트)

### 8.1 검증 흐름 (도구 시작 시)

```
1. .lic 파일 읽기 (없으면 라이선스 등록 화면)
2. Ed25519 서명 검증 (공개키로) → 실패 시 INVALID
3. payload 파싱
4. 현재 시각 가져오기
5. 봉인된 last-seen 비교 → 시계 조작 감지
6. siteId 확인 (Solution Settings 의 사이트 ID 와 비교) → mismatch 시 SITE_MISMATCH
7. expiresAt + graceDays 비교 → 단계 결정 (ACTIVE / IN_GRACE / READ-ONLY / BLOCKED)
8. features[] 와 edition 기준으로 도구 기능 활성/비활성
9. 새 last-seen 시각 봉인하여 저장
```

### 8.2 UI 노출 — Solution Settings › License 카드

```
┌─ License ────────────────────────────────────────┐
│ Edition · enterprise              [ACTIVE]        │
│ Customer · KDB Bank                               │
│ License ID · MPD-2026-0421-KDB                    │
│ Site ID · kdb-corebanking-2026                    │
│ Expires · 2027-04-21 (D-345)                      │
│ Features · multi-site · external-trigger · ...    │
│                                                   │
│ [Show fingerprint] [Upload new license]           │
└───────────────────────────────────────────────────┘
```

만료 단계별로 카드 색이 달라짐:
- ACTIVE → 녹색 점
- IN_GRACE → 호박색 점 + 빨간 배너
- READ-ONLY → 빨간 점 + 강한 경고
- BLOCKED → 거의 빈 카드 + 갱신 요청 안내

## 9. 미결 사항 (Open Questions)

다음은 본 설계에서 *아직 합의되지 않은* 영역. V1 본 개발 착수 전 결정 필요.

| ID | 미결 사항 | 결정 책임 |
|---|---|---|
| OQ1 | 본사 비밀키 보관 방식 — HSM (비용 ↑) vs OS 키링 vs 별도 서버 | 본사 보안 / IT |
| OQ2 | 라이선스 위반 적발 시 본사 대응 절차 (영업 → 법무 → 계약 해지 흐름) | 본사 영업 / 법무 |
| OQ3 | 기능 flag 세분화 — Standard / Pro / Enterprise 각 edition 의 features[] 정의 | 본사 영업 / 제품 |
| OQ4 | 사이트 ID 명명 표준 사내 가이드 — 슬러그 규칙·연도 갱신 시 처리 | 본사 영업 |
| OQ5 | Audit log 의 어떤 시점에 *instance fingerprint* 를 기록하나 — 자동 vs 명시 | 본사 제품 |
| OQ6 | Trial / Demo / Internal 라이선스의 별도 발급 절차 | 본사 영업 |
| OQ7 | sub-license / 위임 발급 옵션 (V1.x 이후 고려) | 본사 영업 / 제품 |
| OQ8 | 라이선스 만료 60일 전부터 갱신 알림을 *고객사* 외 *본사 영업* 에게도 자동 통보하는 채널 | 본사 영업 / SE |

## 11. 다음 Claude 환경에서 이어가는 방법

이 문서는 self-contained. 다른 Claude 환경에서:

1. 본 문서 내용을 메모리에 넣음 (혹은 첨부)
2. 다음 주제 중 하나로 대화 시작:
   - "9번 미결 사항 (OQ1~OQ8) 중 X 를 결정하자"
   - "기능 flag 세분화 표 만들어줘 (OQ3)"
   - "라이선스 발급 도구 (Issuer) UI 설계해줘"
   - "도구 측 License 검증 컴포넌트의 책임·역할 정리해줘"
   - "사이트 ID 명명 가이드 (OQ4) 1페이지로 정리해줘"

본 문서의 "0. 요약" 섹션만 봐도 현 결정 상태가 파악되도록 작성됨. 세부는 각 절에서 참조.
